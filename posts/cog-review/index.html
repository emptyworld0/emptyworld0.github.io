cog 背景介绍

cog 是一款开源的容器化机器学习工具。可以提供容器化的机器学习环境，方便进行模型训练和部署。

简单来讲，使用cog后，可以通过定义一个yaml，通过docker实现推理接口，而不用配置相关推理框架，例如tensorflow、pytorch等。可以实现一行命令调用模型推理，比较方便的调试模型。

网上关于cog的文章较少，从tutorial开始看讲解的比较简单，这篇文章会做一些原理上的介绍。

cog 代码概要

cog的代码分为两部分，一部分是golang，一部分是python。执行make指令后，会分别编译这两部分。其中将golang代码编译成cog二进制文件,将python代码编译成whl。

其中有用到golang 1.16的一个embed特性，将whlembed成[]byte数组，在必要的时候写入到docker中。但为什么要这样设计，表示存疑。其实完全可以用docker的copy命令来替换掉embed。

golang用于编写工具化的部分，python负责运行推理的部分。在AI时代，python的推理地位基本上无法撼动了，我还没有见过其他语言的哪个库实现了tensorflow、pytorch等这些主流的推理库。

推理主体流程图：

  • 首先，需要定义一个cog.yaml文件，该文件指定相关python开发版本，以及相关package版本(本质上是python的requirements.txt文件)。这个阶段有一个py文件比较重要，predict.py。这个文件会定义一个predict函数，该函数会返回一个predictor对象。

  • 其次，运行./cog predict 后，golang编译出来的程序会根据cog.yaml 文件，生成一个dockerfile，以备后续build阶段使用。它使用两段式生成docker的方法，第二个运行阶段会使用第一个阶段的artifact。一个典型的dockerfile如下：

      #syntax=docker/dockerfile:1.4
  FROM python:3.11 as deps
  COPY .cog/tmp/build2428490134/cog-0.0.1.dev-py3-none-any.whl /tmp/cog-0.0.1.dev-py3-none-any.whl
  RUN --mount=type=cache,target=/root/.cache/pip pip install -t /dep /tmp/cog-0.0.1.dev-py3-none-any.whl
  COPY .cog/tmp/build2428490134/requirements.txt /tmp/requirements.txt
  RUN --mount=type=cache,target=/root/.cache/pip pip install -t /dep -r /tmp/requirements.txt
  FROM python:3.11-slim
  ENV DEBIAN_FRONTEND=noninteractive
  ENV PYTHONUNBUFFERED=1
  ENV LD_LIBRARY_PATH=$LD_LIBRARY_PATH:/usr/lib/x86_64-linux-gnu:/usr/local/nvidia/lib64:/usr/local/nvidia/bin
  ENV NVIDIA_DRIVER_CAPABILITIES=all
  RUN --mount=type=cache,target=/var/cache/apt set -eux; \
  apt-get update -qq; \
  apt-get install -qqy --no-install-recommends curl; \
  rm -rf /var/lib/apt/lists/*; \
  TINI_VERSION=v0.19.0; \
  TINI_ARCH="$(dpkg --print-architecture)"; \
  curl -sSL -o /sbin/tini "https://github.com/krallin/tini/releases/download/${TINI_VERSION}/tini-${TINI_ARCH}"; \
  chmod +x /sbin/tini
  ENTRYPOINT ["/sbin/tini", "--"]
  COPY --from=deps --link /dep /usr/local/lib/python3.11/site-packages
  WORKDIR /src
  EXPOSE 5000
  CMD ["python", "-m", "cog.server.http"]
  COPY . /src

  • 再次，dockerbuild完毕后，会执行

    docker run --rm --shm-size 8G --detach --env COG_LOG_LEVEL=warning --publish 0:5000 --mount type=bind,source=/your_cog_project_path/,destination=/src cog-quickstart-base

    将用户定义的目录挂载到docker中，并启动容器。 其中–shm-size 8G 是分配给docker的内存大小，可以根据实际情况进行调整。

  • 最后，通过curl请求，调用docker中的predict函数。

    curl -X POST http://localhost:5000/predict -F image=@image.jpeg

    这里最开始我有个疑问，dockerfile里没有打包predict.py，那么predict函数是如何被调用的呢？

    后来发现，predict函数是在cog.yaml中定义的，在docker启动的时候，predict.py会被拷贝到docker中。通过python的importlib来动态加载使用者自己定义的preict函数。这里如果用golang实现，确实没法实现这么灵活的加载。

以上就是整个的推理流程。

结语

cog 为模型的调试使用，的确很方便。使用者可以自己定义python文件，简单定义完一个yaml后，就可以直接调用模型进行测试了。

但是对于生产环境，cog的使用场景还是比较少的。因为cog只是提供了一个简单的容器化环境，并没有提供其他生产环境所需要的功能。例如，没有提供模型管理、模型部署等功能。

总的来说，cog更适合用于模型调试，而非生产环境。对于了解一个模型推理的0-1过程还是不错的。目前阶段可能实验室的同学用的多一些，生产环境见到用cog的还是比较少的。
